DEPENDÊNCIA TRANSVERSAL — UsuarioResponse.id

O backend `UsuarioResponse` (record Java) não expunha `id`, apenas `login` e
`role`. O dropdown de professor no DisciplinaForm precisa do `id` numérico
para popular o `professorId: Long` do DisciplinaRequest. Sem `id` no
response, ficaria impossível mapear `<option>` → `professorId`.

Solução: adicionado campo `Long id` no record `UsuarioResponse.java` (1ª
posição). MapStruct mapeia automaticamente do `Usuario.id` (entity já tem
`@Id @GeneratedValue`). No frontend, `UsuarioResponse.id` foi declarado
como **opcional** (`id?: number`) porque o `AuthService.getUserFromToken()`
constrói o objeto a partir do JWT (que tem só `sub` e `role`, sem id).

MACRO — Analogia e contexto

Disciplina é a "matéria escolar" — pertence a uma Turma (1 Turma → N
Disciplinas) e pode ter um Professor responsável (FK opcional). No
fluxo escolar real, disciplinas são pré-requisito para Notas e Frequências,
por isso vem antes desses CRUDs na ordem da Fase 3.

O Form precisa de dois dropdowns dinâmicos:
- Turma: obrigatório (validador `Validators.required`).
- Professor: opcional (filtrado de `usuarios` por `role === PROFESSOR`).

Carregamento: `forkJoin({ turmas, usuarios })` paraleliza as duas chamadas
HTTP. Só depois (no callback) é que carrega a disciplina (modo edição).
Alternativa inferior: chamar serial via `concatMap` — desperdiça tempo.

FLUXO DE EXECUÇÃO

  Usuário clica "Nova Disciplina"
        │
        ▼
  abrirNovo() → modalAtivo = 'form', disciplinaSelecionadaId = null
        │
        ▼
  <app-disciplina-form *ngIf="modalAtivo==='form'" [disciplinaId]=null>
        │
        ▼
  ngOnInit → form criado → carregarDropdowns()
        │
        ▼
  forkJoin({ turmas, usuarios }) ──── 2 req HTTP em paralelo
        │
        ▼
  professores = usuarios.filter(u => u.role === Role.PROFESSOR)
        │
        ▼
  Usuário preenche e submit
        │
        ▼
  disciplinaService.save(request)
        │
        ▼
  tap() dispara disciplinaAtualizada$
        │
        ▼
  DisciplinaListComponent.ngOnInit subscribe → carregarDisciplinas()
        │
        ▼
  Lista atualizada automaticamente, modal fecha via close.emit()

MICRO — Destrinchamento

1) `forkJoin({ turmas, usuarios })`
   O QUÊ: combina N Observables; só emite UMA vez quando todos completam.
   POR QUÊ: precisa das duas listas antes de renderizar dropdowns.
   ALTERNATIVA INFERIOR: `combineLatest` — emite a cada update; aqui só
   carregamos uma vez, então `forkJoin` é semanticamente correto.

2) `professores = usuarios.filter(u => u.role === Role.PROFESSOR)`
   O QUÊ: filtra usuarios no client após receber tudo.
   POR QUÊ: backend não tem endpoint /usuario?role=PROFESSOR. Tradeoff:
   trafega usuarios SECRETARIA/ALUNO inutilmente, mas evita criar endpoint
   especializado nesta fase. Refatorar se a tabela usuario crescer muito.

3) `[ngValue]="t.id"` em vez de `value="..."`
   O QUÊ: liga o `<option>` a um valor TypeScript (number), não string.
   POR QUÊ: `value` em select sempre serializa para string; o
   DisciplinaRequest exige `turmaId: number`. `ngValue` preserva o tipo.
   ALTERNATIVA INFERIOR: usar `value` e parsear com `+` na submissão —
   coloca lógica de coerção desnecessária no componente.

4) `professorId: raw.professorId ?? undefined`
   O QUÊ: nullish coalescing — converte null (do form) para undefined.
   POR QUÊ: backend espera `professorId` ausente quando não há professor;
   `null` em JSON também funciona, mas `undefined` é mais idiomático em TS
   para "campo não enviado". Coerente com `professorId?: number` no Request.

5) `id?: number` em UsuarioResponse
   O QUÊ: campo opcional na interface.
   POR QUÊ: AuthService constrói UsuarioResponse a partir do JWT, que não
   tem `id` no payload (sub=login, claim=role, exp). Marcar como
   obrigatório quebraria o construtor do JWT.
   TRADEOFF: quem precisa do id (DisciplinaForm) usa via UsuarioService.
   findAll() — esse retorna do backend com id real.
